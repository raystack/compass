# Roadmap

Compass started as a search and discovery engine for data assets. It stored metadata, indexed it for text search, and let humans find what they needed.

That design assumed humans were the primary consumers of metadata. The world has shifted. AI agents, copilots, and autonomous systems are now the fastest-growing consumers of organizational knowledge. They don't browse a catalog UI. They need a graph they can traverse programmatically, structured context they can reason over, and composable tools they can call.

Compass has evolved into a **context engine** — a type-agnostic knowledge graph that stores entities, relationships, and documents, makes them queryable through multiple interfaces, and lets AI agents compose the context they need.

This document describes where Compass is and where it goes next.

## The Role

Meteor and Compass are two halves of one system. Meteor goes wide — connecting to every source, extracting metadata, and delivering it. Compass goes deep — storing the graph, making it queryable, and serving it to every consumer.

Meteor is the agent that roams your infrastructure. Compass is the brain that remembers what the agent found and answers questions about it.

- **Meteor** owns collection, enrichment, and delivery of raw observations from sources.
- **Compass** owns graph storage, indexing, querying, and serving.

Meteor sends raw observations — "I found this thing at this source with these properties." Compass stores them, constructs the graph, and serves it. This keeps Meteor stateless and simple, while Compass owns persistence and query.

The consumer base spans humans using APIs and AI agents calling tools. Both are first-class. The AI serving path is where most new investment goes.

## Current State

### Entity System

Compass stores **entities** — anything worth naming in an organization's knowledge graph. Tables, services, pipelines, dashboards, people, topics, models, experiments — any non-empty string is a valid type. The type system is open by design.

Each entity carries a URN, type, name, description, freeform properties, and temporal validity (`valid_from`/`valid_to`). Properties are an opaque map — Compass doesn't interpret them. The source system decides what goes in.

### Typed Temporal Edges

Relationships between entities are **typed, directed, temporal edges**. Edge types are open — `lineage`, `owns`, `reads`, `documents`, or any string the ingester provides. Each edge carries properties and temporal validity, enabling point-in-time graph queries.

### Hybrid Search

Search supports three modes, all Postgres-native:

- **Keyword** — tsvector ranking + pg_trgm fuzzy matching
- **Semantic** — pgvector cosine similarity on embeddings
- **Hybrid** — both fused with reciprocal rank fusion (RRF)

No external search engine dependencies. Embeddings are generated asynchronously via a configurable provider (OpenAI, Ollama).

### Documents

Documents (runbooks, annotations, decisions, knowledge) are linked to entities and stored as first-class objects. They flow through the same embedding pipeline for semantic search.

### MCP Server

Compass exposes itself as an MCP server. Any MCP-compatible AI system can connect and get tools:

- `search_entities` — keyword, semantic, or hybrid search across the graph
- `get_context` — entity with its relationships and related entities
- `impact` — downstream blast radius analysis
- `get_documents` — knowledge documents attached to an entity

Tools are composable. An agent investigating lineage calls `get_context`. An agent needing documentation adds `get_documents`. An agent assessing risk adds `impact`. The agent controls what context it assembles — Compass doesn't prescribe bundles.

### APIs

gRPC (Connect RPC) and REST serve the same operations. The MCP server is an additional serving layer for AI agents. Namespace-based multi-tenancy isolates graphs.

## What's Next

### Multi-Hop Graph Traversal

The current graph query model supports single-hop neighborhood lookups and downstream traversal for impact analysis. Real questions require richer traversal:

- "If I change this entity, what breaks downstream?" — multi-hop downstream, filtering by entity type
- "What changed upstream of this entity recently?" — upstream traversal with temporal filters
- "Show the full path from this source to this consumer." — path queries across arbitrary depth

The `get_context` depth parameter exists but is not yet implemented beyond single-hop. The graph query API should support configurable depth, type-filtered traversal, and bidirectional queries. This is foundational — richer graph queries unlock better agent reasoning.

### Entity Resolution

The same logical thing appears across multiple sources — a BigQuery table, a dbt model, a Tableau datasource may all reference the same entity. When Meteor delivers observations from different sources, Compass should match them against the existing graph, recognize duplicates, merge facets, and maintain unified identity.

One entity, multiple source representations, queryable as either. Compass is the right place for this because it has the full graph context needed to make resolution decisions.

### Graph-Aware Ranking

Search ranking today is based on text and semantic relevance. Entities that are highly connected — many dependents, many consumers, central in the graph — are more important than orphaned entities. Search should factor in graph centrality alongside text and semantic scores.

Centrality computation can be async/batch. The score is one signal among several. This applies to any knowledge graph regardless of what entity types it contains.

### Change Feed

Compass is currently pull-only. AI agents and downstream systems that need to stay current should be able to consume graph mutations as they happen. A streaming API that emits events (`entity.created`, `entity.updated`, `entity.deleted`, `edge.created`, `edge.deleted`) would let consumers react to changes without polling.

This is infrastructure work — back-pressure handling, cursor-based replay, event ordering. Build it when there's a concrete consumer that needs it.

## Priorities

1. **Multi-hop graph traversal** — Make the depth parameter work. Add type-filtered, bidirectional, and path queries. This is the highest-leverage improvement to the graph query layer.

2. **Entity resolution** — Deduplicate and merge entities from multiple sources. This is core to Compass's value as the single graph of record.

3. **Graph-aware ranking** — Factor connectivity and centrality into search result ranking. Improves result quality for both humans and agents.

4. **Change feed** — Streaming graph mutations for reactive consumers. Build when there's a real consumer that needs it.

## What Stays the Same

- **Postgres as the source of truth.** All storage and search is Postgres-native — entities, edges, documents, text search (tsvector, pg_trgm), and vector search (pgvector). No external dependencies.
- **gRPC and REST APIs.** Existing integrations keep working. MCP is an additional serving layer.
- **Open entity model.** Any type, any properties, any edge type. Compass doesn't interpret domain semantics — it stores and serves the graph.
- **Namespace-based multi-tenancy.** The isolation model carries forward.
- **Composable tools over monolithic endpoints.** AI agents assemble the context they need by calling focused tools. Compass doesn't prescribe what context looks like.
